# 収支管理アプリ 要件定義書 v3.2

作成日: 2026-04-28
オーナー: ふりすく
変更点: 支出を性質別に3分類(期限なし定期/期限あり定期/単発)。
        定期支出はマスタ管理(予測テーブル不要)、単発支出のみ予実管理。
        定期支出の自動実績化ロジック、サブスク一覧画面を追加。

---

## 1. 全体像とスコープ

### 1.1 最終ゴール
**自分が知りたいことを1つに集約するパーソナルダッシュボード**を構築する。
対象領域は以下の3つ:

1. 収支管理(支出・収入・残預金の予実)
2. ソシャゲ管理(課金・進捗・履歴)
3. 株の履歴管理(銘柄・買付理由・振り返り)

### 1.2 本ドキュメントのスコープ
**本ドキュメントは「収支管理」MVPの要件のみを定義する。**
ソシャゲ管理・株管理は将来フェーズとして概要のみ記載(§10)。

---

## 2. 目的

### 2.1 主目的
月次の**残預金**を「予定 vs 実績」で管理し、**貯蓄可能額**を把握・予測する。

### 2.2 性質
- **把握型**: 何にいくら使ったか・何で稼いだかを可視化する
- **計画型**: 来月以降の残預金を予測し、貯蓄可能額を見積もる

### 2.3 非目的(やらないこと)
- 抑制型(予算オーバーアラート、強制制御等)はMVPに含めない
- リアルタイム残高反映(銀行APIスクレイピング等)は対象外
- 日次の残高推移管理(月末残高のみ管理する)

## 3. ★収入要件(v3:予実管理対応)

### 3.1 収入カテゴリマスタ
- 収入のカテゴリをユーザーが管理する(CRUD可能)
- 初期セット例:
  - **給与**(必須)
  - 賞与(必要に応じて追加)
  - 副業 / その他(必要に応じて追加)

### 3.2 収入の予実管理

支出と同じ構造で予測テーブル・実績テーブルを分離する。

#### 収入_予測テーブル
- 単位: **年月 × 収入カテゴリ × 金額**
- 例: 2026-05 / 給与 / 300,000円
- 入力方法: ユーザーが手動でセット
- **給与の予測値は固定額**(残業代・賞与等の変動分は含めない、保守的見積もり)

#### 収入_実績テーブル
- 単位: **個別の入金記録**
- 必須項目: 入金日 / 金額 / 収入カテゴリ
- 入力方法: 手動入力 / インポート
- **実績は変動分込みの実額**(残業代等が含まれた振込額そのまま)

#### 表示時の切替ロジック(★カテゴリ単位)
当月の収入をカテゴリごとに集計する際、**カテゴリ単位**で予測/実績を切り替える(支出と同じ思想)。

```
カテゴリTの当月収入:
  実績テーブルにそのカテゴリの当月データがある?
    YES → 実績の合計を使う(実モード)
    NO  → 予測テーブルの値を使う(予モード)
```

### 3.3 給与の入金日
- **毎月25日**に入金される
- 25日が土日の場合は前営業日(または翌営業日)に入金(運用に合わせる)
- 残高予測ロジックではこの日に給与額が加算される

### 3.4 予測 vs 実績の差分の意味
- 給与の予測 = 固定額、実績 = 変動込み
- **差分 = 残業代等の変動収入**として自然に表現される
- 別レコードを作る必要なく、予実差分で見える

## 4. ★支出要件(v3.2:性質別に再構成)

### 4.1 支出の3分類

支出は性質別に3つに分類して管理する。

| 分類 | 性質 | 例 | 管理方法 |
|---|---|---|---|
| **A. 期限なし定期支出** | 月額固定、終わりなく続く | 親への送金、保険 | 本体マスタ + 実績テーブル |
| **B. 期限あり定期支出(サブスク)** | 月額固定、解約しうる | Netflix、Spotify、ChatGPT Plus | 本体マスタ + 実績テーブル |
| **C. 単発支出(変動費)** | その都度発生、金額バラバラ | 食費、ソシャゲ、Amazon、雑費 | 予測テーブル + 実績テーブル |

**A・Bはマスタ自体が金額の確定値を持つため、予測テーブルは不要。**
**Cは金額がブレるため予測テーブルで予測値を持つ。**

### 4.2 カテゴリ構造(主にC=単発支出向け)

C(単発支出)は**2階層**でカテゴリ管理する。

```
大カテゴリ(予実で見る粒度)
  └ 詳細カテゴリ(分析時のドリルダウン用)
```

| 大カテゴリ | 詳細カテゴリ例 |
|---|---|
| ソシャゲ | すたれ / めいちょう / エンフィ / ゼンゼロ / アークナイツ |
| 食費 | 自炊(スーパー) / Uber Eats / 出前館 / 外食 |
| 雑費 | Amazon / 日用品 / その他 |
| その他 | (用途別に追加) |

A・B(定期支出)は分類タグとして「サブスク」「口座引落(送金/保険)」などを持つ。

### 4.3 カテゴリ管理
- 大カテゴリ・詳細カテゴリともに**ユーザーが追加・編集・削除可能**
- カテゴリ削除時は既存データの整合性を保つ(過去データは保持、または「未分類」へ移動)

### 4.4 A: 期限なし定期支出マスタ

#### マスタ項目
- 名前(例: 親への送金、生命保険)
- 月額
- 引落日(毎月◯日)
- 支払方法(口座引落 / 現金/振込 など)
- 開始日
- 備考

#### 動作
- アプリ起動時に「**今日までに過ぎた引落日**」をチェック
- 該当するマスタは実績テーブルに自動でレコード生成
- 月額が変わった場合は最新値で上書き(履歴は持たない)

### 4.5 B: 期限あり定期支出マスタ(サブスク)

#### マスタ項目
- 名前(例: Netflix、Spotify)
- 月額
- 更新日(毎月◯日)
- 支払方法(PayPayカード / Amazonカード など)
- 開始日
- **終了日(null可、解約時にセット)**
- 備考

#### 動作
- アプリ起動時に「**今日までに過ぎた更新日**」をチェック
- ただし**終了日が過ぎているものは対象外**
- 該当するマスタは実績テーブルに自動でレコード生成
- 月額が変わった場合は最新値で上書き(履歴は持たない)

#### サブスク一覧画面
| 名前 | 月額 | 更新日 | 支払方法 | 状態 |
|---|---|---|---|---|
| Netflix | ¥1,490 | 15日 | PayPay | 有効 |
| Spotify | ¥980 | 1日 | Amazon | 有効 |
| ... | ... | ... | ... | ... |
| **月額合計** | **¥X,XXX** | | | |

- 解約済み(終了日が過去)は別タブまたはフィルタで非表示にできる
- 月額合計が一目で分かる(サブスク見直し用)

### 4.6 C: 単発支出の予実管理

**Cのみ予測テーブル・実績テーブルの両方を持つ。**

#### 予測テーブル(支出_予測)
- 単位: **年月 × カテゴリ × 金額**
- 例: 2026-05 / 食費 / 30,000円
- 入力方法: ユーザーが手動でセット(直近Nヶ月平均から自動初期値も可)

#### 実績テーブル(支出_実績)
- 単位: **個別の支出記録**
- 必須項目: 発生日 / 金額 / 大カテゴリ / 詳細カテゴリ / 支払方法
- 入力方法: 手動入力 / インポート(JSON/CSV)

#### 表示時の切替ロジック(カテゴリ単位)
当月の支出をカテゴリごとに集計する際、**カテゴリ単位**で予測/実績を切り替える。

```
カテゴリXの当月支出:
  実績テーブルにそのカテゴリの当月データが1件でもある?
    YES → 実績の合計を使う(実モード)
    NO  → 予測テーブルの値を使う(予モード)
```

### 4.7 実績テーブルの一本化

A・B・Cの実績はすべて**同じ実績テーブル**に格納する。
- A・B由来: マスタから自動生成されたレコード(発生日 = 引落日/更新日)
- C由来: 手動入力 or インポートで登録されたレコード

これにより残高計算は実績テーブル1本を見れば済む。

#### 動作イメージ(5月途中の状態例)
```
A(期限なし定期):
  親への送金   : 5/10引落予定 → 5/10過ぎていれば実績、未来なら未生成
  生命保険     : 5/27引落予定 → 5/27過ぎていれば実績、未来なら未生成

B(サブスク):
  Netflix      : 5/15更新 → 5/15過ぎていれば実績、未来なら未生成
  Spotify      : 5/1 更新 → 既に実績生成済み

C(単発):
  食費         : 実績データあり → 実モード(実績合計)
  ソシャゲ     : 実績データなし → 予モード(予測値)
  雑費         : 実績データなし → 予モード(予測値)
```

## 5. 支払方法と支払いサイクル

### 5.1 支払方法の種類
| 支払方法 | 締め日 | 引落日 | 残高への影響 |
|---|---|---|---|
| **PayPayカード** | 月末 | 翌月27日(土日なら翌営業日) | 引落日にまとめて減る |
| **Amazonカード** | 月末 | 翌月27日(土日なら翌営業日) | 引落日にまとめて減る |
| **口座引落** | 契約ごと | 契約ごと(月内の固定日) | 引落日に減る |
| **現金/デビット** | (即時) | 発生日 | 発生日に減る |

### 5.2 各支出は「支払方法」を必ず持つ
実績テーブルの各レコードに支払方法を記録する。
これにより、**いつ口座から金が出ていくか**が一意に決まる。

### 5.3 カード請求の集計
カード(PayPay/Amazon)の利用分は**カード締め月単位**で集計する。

```
5/27の口座からのカード引落額
  = PayPayカードの4月利用合計
  + Amazonカードの4月利用合計
```

カードA/Bは内部的には区別して記録するが、残高予測上は**合算**して扱う。
(カード別の利用分析は副次機能として可能)

### 5.4 口座引落(保険など)の管理
- サブスクと同様に**月額固定の引落リスト**として登録
- 各レコードに「**引落日(毎月◯日)**」を持たせる
- 残高予測ではその日に該当額が減る扱い

## 6. 残預金管理

### 6.1 対象口座
- **みずほ銀行のみ**
- 楽天銀行は証券口座と紐付いているため対象外(投資領域に分離)

### 6.2 残高入力
- **月末残高**のみ手動入力(月1回)
- 初回利用時のみ「初期残高」を別途入力
- 月末残高は翌月の月初残高として扱う

### 6.3 残高表示ロジック
| 状態 | 表示する値 |
|---|---|
| その月の月末実残高がDBに存在する | **実残高(確定値)** |
| その月の月末実残高がDBに存在しない | **予測残高(計算値)** |

### 6.4 ★予測残高の計算式(v3.2:支出3分類対応)

**支払日ベース**で口座の入出金を計算する。月末残高のみ予測する。

```
当月末の予測残高
  = 月初の実残高
  + 当月の収入合計(カテゴリごとに予/実を判定して合算)
    例: 給与の25日入金分(=実 or 予)
        その他収入(あれば)
  − 当月中に発生する現金/デビット支出
  − 当月27日のカード引落額(= 前月のカード利用合計)
  − 当月中の口座引落額(A/B/Cの口座引落分すべて)
```

#### 集計の元データ
- **収入**: 当月の予実(カテゴリ単位で判定)
- **C(単発)の現金/デビット支出**: 当月の予実(カテゴリ単位で判定)
- **C(単発)のカード支出**: 前月実績合計 → 当月27日にカード引落で出ていく
- **A(期限なし定期)**: マスタの月額 × 当月該当分(引落日に減る)
- **B(サブスク)**: マスタの月額 × 当月該当分(更新日 → カード or 口座経由で減る)

#### 重要:A・Bは予測テーブル不要
A・Bはマスタが金額を持つため、予測値=マスタ値で確定。
当月の更新日が来たら自動で実績化され、残高計算に反映される。

## 7. 予実管理ロジック(統合)

### 7.1 ローリングの考え方
- **過去月**: 月末実残高(=確定値)を表示
- **当月**: 収入カテゴリ・支出カテゴリそれぞれで予/実が混在し、合算した予測残高を表示
- **未来月**: 全カテゴリ予測ベースで予測残高を表示

### 7.2 予測値の元データ
予測テーブルの値は以下の優先順位で初期化される(ユーザー上書き可能):

1. **直近Nヶ月の実績平均**(初期値はN=3、設定可能)
2. データ不足時はユーザー手入力
3. 何もなければ0

ユーザーが手動でセットした予測値は自動計算で上書きされない。

### 7.3 収入予測の特例
- **給与は固定額**で予測する(平均ではなく契約上の額面)
- これにより、残業代等の変動分が「予実差分」として可視化される
- その他の収入カテゴリ(賞与、副業等)は通常通り平均または手動入力

## 8. データ取込み

### 8.1 アーキテクチャ方針
**Claude経由でのインポート方式**を採用する。

```
[データソース] → [Claude(解析・整形)] → [JSON/CSV] → [アプリ(インポート)]
```

理由:
- アプリ本体に解析ロジックを持たせない(=軽量化・保守容易)
- メール形式変更等への追従はClaudeスキル側で対応可能
- 既存の `sosha-gmail-update` スキルと方式を統一できる

### 8.2 データソース一覧
| ソース | 取得方法 | 主な対応カテゴリ |
|---|---|---|
| Google Play | Gmail解析(Claude) | 支出: ソシャゲ |
| Amazon | Gmail解析(Claude) | 支出: 雑費 |
| Uber | Gmail解析(Claude) | 支出: 食費 |
| 出前館 | Gmail解析(Claude) | 支出: 食費 |
| カード明細 | CSV解析(Claude) | 各種(支払方法の確定) |
| サブスク | 手動登録(固定値) | 支出: サブスク |
| 口座引落(保険等) | 手動登録(固定値) | 支出: 専用 |
| 給与振込 | 手動入力 or 銀行明細CSV | 収入: 給与 |

### 8.3 実装の優先順位(MVP)
| フェーズ | 対象 | 理由 |
|---|---|---|
| **第1弾** | Google Play 自動取込み + その他は手動入力 | 既存スキル流用、ソシャゲ課金は金額影響大 |
| 第2弾 | Amazon / Uber / 出前館 のメール解析 | 利用頻度が高い |
| 第3弾 | カード明細CSV取込み・銀行明細CSV取込み | 月次バッチで対応 |

### 8.4 想定運用フロー
```
日常: 必要に応じて手動で支出を記録(または何もしない)

月末:
1. ふりすくがClaudeに「今月分まとめて」と依頼
2. ClaudeがGmail/CSVを解析し、JSON/CSVを出力
   - 各レコードに「支払方法」を含める
3. アプリでインポート
4. みずほ月末残高を手動入力
5. 給与の実績額を入力(残業代等を含む実額)
→ 月次の予実が確定し、翌月予測が更新される
```
