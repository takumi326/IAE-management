# 収支管理MVP 整合レビュー (v1)

## 対象

- 要件定義: `ignore/収支管理MVP_要件定義書_v3_2.md`
- DB設計: `ignore/収支管理MVP_DBスキーマ設計書_v3.md`
- API設計: `ignore/収支管理MVP_API設計書_v1.md`
- UI設計: `ignore/収支管理MVP_UI設計書_v2.md`

## 結論

実装を開始する前に、**要件定義(v3.2)とDB/API/UI(v3/v2/v1)の間で運用粒度が異なる**ため、MVP方針を固定する必要がある。

現在の実装ドキュメント群(DB/API/UI)は、以下の方針で揃っている:

- 予想は月単位合計のみ
- 実績は月単位で保持
- 日次引落計算はしない
- 定期支出の自動実績化はしない

## 主要な差分

### 1) 予想粒度

- 要件: カテゴリ単位で予実切替 (収入/支出とも)
- DB/API/UI: 月単位の収入合計・支出合計のみ

影響:
- カテゴリ別の予想精度・比較機能はMVPから外れる

### 2) 日付粒度

- 要件: 引落日/更新日/発生日ベースを重視
- DB/API: `year_month` (月初日)のみで扱う

影響:
- 日次キャッシュフロー予測は不可
- 「当月27日にカード引落」を厳密に再現しない

### 3) 定期支出の自動実績化

- 要件: 起動時に定期支出を自動実績化
- DB/UI: その機能を非対応扱い

影響:
- 実績入力は手動またはインポート依存

### 4) モデル命名の表現揺れ

- DBは日本語論理名中心
- APIは英語snake_case中心
- UI文言は日本語

影響:
- 実装時にマッピング表を作らないと混乱しやすい

## 実装で採用する確定方針 (提案)

MVPはDB/API/UIの現行方針に合わせる:

1. 月次管理を最優先し、日次管理は将来フェーズ
2. 予想は `forecasts` テーブルで `income/expense` の2系統のみ
3. 定期支出の自動実績化は実装しない
4. 要件書上のF-06/F-08は「将来拡張」に明示的に移管
